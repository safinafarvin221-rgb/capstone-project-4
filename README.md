# Telecom Churn Prediction – Part 4 (LLM Powered Feature)

## Chosen Track

**Track C – Model Prediction Explanation Pipeline**

This project combines a Machine Learning model with a Large Language Model (LLM) to provide human-readable explanations for churn predictions. The trained ML model predicts whether a customer is likely to churn, while the LLM generates a structured JSON explanation based on the prediction result.

---

## Project Overview

The objective of this project is to combine predictive analytics with Generative AI.

The pipeline performs the following steps:

* Load the trained ML model (`best_model.pkl`)
* Create three customer feature inputs
* Predict churn using `.predict()`
* Calculate churn probability using `.predict_proba()`
* Construct an LLM prompt using prediction details
* Generate a structured JSON explanation
* Validate the JSON response
* Apply a PII safety guardrail before every LLM request

## LLM API Connection

The API key is stored securely in a **`.env`** file and loaded using environment variables.

Example `.env` file:

```env
LLM_API_KEY=YOUR_API_KEY_HERE
```

The notebook loads the key from the environment instead of hardcoding it:

```python
from dotenv import load_dotenv
import os

load_dotenv()

api_key = os.getenv("LLM_API_KEY")
```

The actual API key is never committed to the repository.

## Why Temperature = 0?

Temperature controls randomness in LLM output.

Temperature 0 was selected because this task requires:

* Deterministic output
* Consistent JSON formatting
* Predictable business explanations
* Easier validation

Higher temperatures introduce randomness and may produce different explanations for the same input.

---

## Temperature Comparison

| Input    | Output at temp=0            | Output at temp=0.7          | Key Difference     |
| -------- | --------------------------- | --------------------------- | ------------------ |
| Sample 1 | Consistent JSON explanation | More variation in wording   | Higher creativity  |
| Sample 2 | Stable explanation          | Different explanation style | Less deterministic |
| Sample 3 | Same structure every run    | More varied wording         | Higher randomness  |

Temperature 0 always chooses the highest-probability token, producing deterministic and repeatable outputs.

Temperature 0.7 samples from a wider probability distribution, producing more diverse responses.

## Structured JSON Schema

Expected response:

```json
{
    "prediction": "string",
    "probability": "number",
    "reason": "string",
    "risk_level": "string",
    "recommendation": "string"
```

Required fields:

* prediction
* probability
* reason
* risk_level
* recommendation

Validation process:

* Parse response using `json.loads()`
* Validate using `jsonschema.validate()`
* If validation fails, return a fallback dictionary.

## PII Guardrail

Before every LLM request, user input is checked for Personally Identifiable Information (PII).

Regex checks include:

* Email addresses
* Phone numbers

Function:

```python
has_pii(text)
```

If PII is detected:

```
Input blocked: PII detected.
```

The LLM request is not executed.
